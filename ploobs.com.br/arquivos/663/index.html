Software Render no XNA


Olá Pessoal, neste post irei apresentar um projeto que implementei recentemente. Trata-se de um Software Render Completo que realiza todas as transformações geométricas discutidas no post http://ploobs.com.br/?p=622 sobre Renderização 3D.

O papel do renderizador é converter uma descrição de uma cena em uma imagem na tela. Os principais passos conforme discutidos no post anterior são: (estamos trabalhando com renderização baseada em rasterizacão, existem outras maneiras de renderizar uma cena como raytracing e radiosity):

  1. Geração da Geometria
  2. Posicionando os Objetos na Cena
  3. Posicionando e Configurando a Câmera
  4. Clipping e Rasterização
  5. Colorir os Pixels
  6. Alpha Blending, Alpha, Stencil e Depth Tests

A seguir descreverei cada umas destas fases novamente, porém desta vez sob a perspectiva do render. (falando de como foi implementado =P, algumas partes mais para a frente exigem um conhecimento meio avançado de computação gráfica, qualquer dúvida, pergunte no nosso fórum ou no espaco de comentários)

Geração da Geometria

Esta fase normalmente não está conectada ao Render. Ela é realizada por um software de modelagem 3D como o 3D Max Studio. O Render recebe apenas um arquivo descrevendo a geometria e texturas da cena.

O software render construído suporta o formato .X, o .FBX, e um formato XML que eu criei um tempo atrás em um outro projeto http://modelxml.codeplex.com/ (no futuro prentendo fazer um post sobre ele). Ao receber um modelo/cena, o software render apenas o converte para um formato interno mais otimizado  (Um fator interessante é que devemos “alinhar” o modelo importado com o nosso sistema de coordenadas interno (usamos mão direita com Y para cima semelhante ao DirectX), isto influencia como as matrizes View e Projection são geradas, o render construído espera um modelo semelhante ao XNA/DirectX e não efetua nenhuma conversão).

Posicionando os Objetos na Cena   e    Posicionando e Configurando a Câmera

O posicionamento de objetos consiste em criar a Matriz World de cada objeto da cena, o usuário pode setá-la manualmente (criando uma Matrix contendo uma componente de rotacão, uma de translacão e uma de escala) ou importá-la do 3DS Max (Local Transform do objeto carregado).

O posicionamento e configurarão da câmera consiste em setar as Matrizes View e Projection. A View nada mais é do que uma matriz de mudança de base que leva a cena para o sistema de coordenadas definido pela posição e pelos vetores Up e Target da câmera (com estas três informações podemos criar uma base ortonormal e em seguida criar a matriz de mudança de base entre a canônica e esta recém definida). Esta matriz é criada pelo XNA/DirectX usando a famosa função Matrix.CreateLookAt (quem tiver curiosidade, qualquer livro de computação gráfica/Algebra Linear ensina como cria-la).

A Projection consiste em uma matrix que realiza projeção em perspectiva (pode-se usar projeção ortográfica, porém o resultado é pouco real). Sua base teórica é a famosa e famigerada SEMELHANÇA DE TRIÂNGULOS. No caso do render eu usei a função do XNA Matrix.CreateProjectionMatrix para realizar este trabalho pra mim (qualquer livro de comp gráfica mostra como cria-la).

Ao invés de desenvolver um software render estático e otimizado eu preferi criar uma versão em software da GPU programavel. Implementei em software as fases InputAssembler,VertexShader,Rasterizer,PixelShader e OutputMerger (veja um artigo que eu fiz sobre a pipeline da GPU http://www.pontov.com.br/site/index.php/xna/70-shaders/250-introducao-a-shaders). (Do ponto de vista pratico, o Software render construído literalmente não serve para nada >.<, porém do ponto de vista teórico e didático ele é excelente =P, pois mostra exatamente o que ocorre de verdade na nossa GPU). Uma consequencia interessante desta abordagem é que podemos debugar e programar os “shaders” usando C# e .Net.

O render construído implementa parcialmente o pipeline Direct11 (com os seguintes estágios programaveis: Vertex Shader, Geometric Shader, Hull Shader, Domain Shader, Pixel Shader).

A figura abaixo mostra as fases que foram implementadas (Omiti as partes específicas do DirectX 11, pois complicaria desnecessariamente o desenho e a compreensão)

Implementacao em software da GPU

 

Os quadrados sao blocos configuráveis e os hexágonos sao programáveis em C# (basta extender uma interface e fazer o seu próprio shader =P)

O bloco InputAssember pode receber primitivas indexadas (com vertex e index buffer) ou primitivas normais (sem index buffer),  além disto, o render pode receber pontos ou triângulos como tipos de primitivas. O papel deste bloco é apenas extrair cada um dos vértices corretamente do vertexbuffer (lembre que vertice nao é apenas posicao, ele pode conter coordenadas de textura, normais ….) e enviar individualmente cada um para o VertexShader (numa pipeline real, teríamos varias unidades de VertexShader, no nosso caso temos apenas um (single threaded)).

O vertex shader (interface que o usuário implementa e fornece para o render) é chamado uma vez para cada vértice de cada triângulo do modelo. A saída desta fase é também um vértice (que pode ou nao estar em espaco de projecao). É  responsabilidade do criador do shader transformar os vértices para o espaco de projecão, as vezes como por exemplo quando desenhamos full screen quads, os vértices passados já estão no espaco correto e não precisam de transformacão. O render oferece algumas funcões idênticas as intrínsicas da HLSL (como dot,reflect,normalize … ).

Os vertices que saem do VerticeShader vão para o Primitive Assember que apenas junta-os  e forma as primitivas corretas (Ex: junta os três vertices de um triângulo e o envia para a próxima fase)

A fase Geometric Shader é opcional, alguns shaders não a utilizam, caso ela não exista a primitiva criada vai direto para o Rasterizer. (Geometric Shader é um estágio programável inserido no Shader Model 4 – DirectX 10 ).

O Geometric Shader recebe como entrada uma primitiva (3 vértices que saíram do Vertex Shader no caso de triângulos) e os manipula (aplica transformações por exemplo ). É possível criar/destruir primitivas (podemos ate receber triangulos do vertex shader e enviar pontos para o Rasterizer).

A funcionalidade conhecida como Stream Out no DirectX 10 e 11 tambem foi implementada. O usuário pode optar por enviar as primitivas que saíram do GeometricShader ou Vertex Shader para fora da “GPU” sem passar pelos estágios de rasterizacao , Pixel Shader …. Estas primitivas são enviadas “para fora” como um VertexBuffer bastante semelhante ao DirectX.

Clipping e Rasterização

Esta fase recebe as primitivas em espaco de projecao (que vieram do Vertex ou Geometric Shader) e efetua os processos de BackFaceCulling, Clipping e Rasterizacão.

O atributo position de um Vértice em espaco de projeção é um vetor X,Y,Z,W sendo que a componente W é o componente homogeneo. Para recuperar a posição correta do vertice na tela (em coordenadas de tela normalizadas) devemos dividir o X,Y,Z por W, porem este W é usado na rasterizacao para fazer correcão de perpectiva. Desta forma o Vertex Shader (ou Geomtric Shader se ele estiver presente) envia para o rasterizador a position sem dividir pelo W. (Ao multiplicar uma position pela matriz Projection estamos criando este X,Y,Z,W, com W diferente de 0 – esse “W” é transparente para o usuário comum) O processo é bastante semelhante ao que acontece em HLSL nas GPUs de “verdade” )

O Rasterizer inicialmente efetua o Clipping (no nosso caso nao é um clipping verdadeiro, está mais para culling, estamos apenas verificando se as primitivas em espaco de projecao estao COMPLETAMENTE dentro do Frustrum  também em espaco de projecao (que vira uma Box no caso da nossa matriz projection), ou seja, checa-se se as coordenadas de posicao divido pelo componente homogeneo (X/W,Y/W,Z/W) de cada vertice das primitivas estão entre -1 e 1 e se não estiverem a primitiva é removida — pois não será vista pela câmera).

Após o clipping ter sido feito, efetua-se o back face culling (apenas ativada em caso de triângulos). Este é bastante simples, basta calcular a normal de cada triângulo (cria-se dois vetores utilizando os 3 vértices e calcula-se o produto vetorial entre eles) e verificar o ângulo que ela faz com o vetor UnitZ (em espaco de projecao, o vetor UnitZ aponta para dentro da tela, se a normal do triangulo estiver apontando para esta direcao logo estamos vendo apenas a parte deste triângulo, então podemos descartá-lo). Podemos setar qual a ordenacao dos vertices dos triângulos usados (Clockwise e CounterClockWise).

Antes de ir para a rasterizacão, convertermos as posições dos vertices que estão em espaco de projecao normalizado para coordenadas de tela (os X,Y estão entre -1 e 1, convertemos eles para o range 800,600 por exemplo)

Os triangulos remanescentes irão para a rasterizacão. O algoritmo implementado foi o Triangle Baricentric Rasterization (lembre-se que pontos nao sao rasterizados nem sofrem back facing culling). Este paper explica por cima como que o processo é feito http://www.cs.caltech.edu/courses/cs171/barycentric.pdf (sem correcao de perspectiva)

  1. Criamos um bounding rectangle em volta do nosso triângulo a ser processado (que está em coordenadas de tela normalizada)
  2. Discretizamos esse bounding rectangle (supondo que o bounding seja representado pelos pontos (0,0) e  (50,50) — ponto cima-esquerda e baixo-direita, entao percorremos com dois For alinhados cada pixel deste bounding como o exemplo a seguir: (0,1), (0,2) … (0,50) … (1,0),(2,0) … (50,0) … (1,1), …)
  3. Para cada “pixel” gerado,  verificamos se ele esta ou nao dentro do triangulo processado (usando coordenas baricentricas esta tarefa é trivial, basta calcular o alpha, beta e gama da representacao baricentrica e verificar se eles estao entre 0 e 1). Os pixels que satisfizerem esta condição são enviados para o Pixel Shader, o outros são rejeitados
  4. Para cada pixel aceito interpola-se os atributos dos vértices do triangulo em questao usando as coordenas baricentricas como fator interpolador (veja o paper, é apenas uma média ponderada =P). Em seguida aplica-se a correcao de perspectiva. (Matematicamente não podemos interpolar as coordenadas de texturas pois elas nao variam linearmente, entao interpolamos o 1/W (componente homogeneo)  e o usamos para gerar os valores correctos de U e V veja aqui uma explicação interessante http://www.gamedev.net/topic/592675-perspective-interpolation/ sobre o assunto)

Caso voce nao tenha entendido nada do algoritmo anterior, tente ler o paper citado ou olhar o codigo fonte (se ainda estiver com duvida depois disso pergunte =P).

Colorir os Pixels

O Pixel Shader recebe os pixels gerados e os colore (o usuario deve extender a interface PixelShader para isso). Uma maneira clássica de realizar esta tarefa é ler a cor de uma textura. O render implementa Bilinear Filtering (http://en.wikipedia.org/wiki/Bilinear_filtering) para aliviar um pouco os problemas de Magnification e Minification envolvidos na leitura de texels. Nao implementei MipMap >.< (a idéia é bastante simples no caso de alguem querer se aventurar)

Alpha Blending, Alpha, Stencil e Depth Tests

Por fim os pixels passam pelo Alpha Blending (desta vez eu apelei e nao segui o Directx, simplesmente aceito um delegate (algo como um ponteiro de funcão) que decide como combinar o pixel que esta “chegando” da pipeline com o pixel que ja se encontra no Buffer de saída). Stencil Test não foi implementado.

O clássico algoritmo de Depth Test foi implementado usando uma estrutura que armazena as distâncias (Z/W -> entre o “pixel” e a câmera em espaco de projecao) . Caso chegue um pixel que tenha uma distancia menor que a armazenado no DepthBuffer para a mesma posição de tela, este pixel é armazenado no Buffer de saida, caso contrario é descartado. (Dá para configurar algumas propriedades deste estágio no render).

Foi implementado o Conceito de Render Target, é possível renderizar um modelo em uma texture e depois utilizar esta textura para colorir um segundo modelo.

Os pixel que sobreviveram a TODOS os testes anteriores estarão no buffer de saída (que no fundo não passa de um array de cores). A cada frame simplesmente copiamos o seu conteudo dele para a tela (desta vez utilizando a GPU de verdade =P)

Implementação e Download

O Software Render foi implementado em C# e XNA 3.1 e pode ser baixado aqui http://softrender.codeplex.com/

Recomendo Fortemente que o pessoal interessado em criar shaders para a GPU de “verdade” de uma olhada e uma modificada no render construído. Ele vai auxilia-lo a entender PROFUNDAMENTE como que as transformação geométricas realmente ocorrem nos bastidores da GPU.

Umas das features interessantes do Software Render que não foi discutida com profundidade é o Tessellation. Criei (na verdade foi uma mistura de alguns existentes mais uma leve contribuição minha) um algoritmo de fractional Tessellation com resultados bastante semelhantes ao obtido com o DirectX 11 (OBVIAMENTE com uma velocidade trilhões de vezes menor).
