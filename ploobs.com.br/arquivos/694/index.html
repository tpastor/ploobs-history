CPU VS GPU – Comunicação Assíncrona — Pt 1



Apesar de transparente para o usuário, muitas vezes é importante conhecer um pouco dos mistérios sobre como ocorrem as trocas de informações entre o processador e a placa de vídeo. Este post irá falar um pouco de como isto ocorre com DirectX/XNA.

A CPU e a GPU funcionam de maneira completamente assíncrona e independente, para compreender como que a comunicação entre estas duas entidades funciona é bastante importante compreender o papel de cada uma em um cenário típico de um jogo.

CPU

  • Atualizar Física (mesmo em sistemas como o Physx, a CPU ainda executa grande parte do trabalho), Estado interno dos objetos, processar regras do jogo, Inteligência Artificial, …
  • Traduzir as chamadas de desenho em um formato que o Driver de vídeo possa entender (Ex: A CPU (DirectX Runtime) precisa converter as as chamadas de desenho como SpritBatch.Draw do XNA ou Device.DrawUserPrimitive do DirectX em instrucões que o Driver de vídeo).
  • Converter e Enviar estas instrucões para a GPU (o Driver de vídeo converte as informacões passadas em instrucões para a GPU).

GPU

  • Efetuar as ordens dadas pela CPU, ou seja, por exemplo rodar o código de um shader e renderizar um modelo. (estou desconsiderando GPGPU aqui)

Estas operações (Lado CPU e Lado GPU) são feitas em paralelo conforme mostra a figura abaixo.﻿

Funcionamento Assincrono

É importante salientar que em um determinado frame são feitas diversas chamadas de desenho de primitiva como a mostrada a seguir:

 g_d3d_device->SetStreamSource(0,                   //StreamNumber
                                 g_list_vb,           //StreamData
                                 0,                   //OffsetInBytes
                                 sizeof(tri_vertex)); //Stride

   g_d3d_device->DrawPrimitive(D3DPT_TRIANGLELIST, //PrimitiveType
                               0,                  //StartVertex
                               g_list_count);      //PrimitiveCount

 

(DirectX – desenha uma serie de triangulos contidas em um vertex buffer):

Porém, na realidade, nenhuma primitiva esta sendo desenhada de fato e NENHUM dado está de fato sendo enviado para a GPU no momento em que estas funcões estão sendo chamadas.

O que ocorre na realidade é o seguinte processo (em DirectX 9c e Xna):

  • Ao chamar o método de desenho (como o mostrado acima), as ordens gráficas são guardadas em um buffer na CPU.
  • Todos os métodos de desenho terminam (Em XNA é o momento em que termina a funcão Draw).
  • Caso XNA -> O framework internamente chama GraphicsDevice.Present.
  • Este método chama o método nativo Directx IDirect3DDevice::Present (Se voce usar DirectX, voce chamara o metodo Present manualmente)
  • A DirectX Runtime (que é uma COM) converte as chamadas que estão no buffer citado em chamadas para o driver gráfico(Nvidia nview.dll  por exemplo), e em seguida manda o driver executar as tarefas passadas.
  • O driver converte novamente as informações para um formato que a GPU entenda e envia os comandos para ela (o seu shader hlsl vira uma coleção de Bytes por exemplo).
  • A CPU continua o seu processamento normal, devolvendo o controle para a função IDirect3DDevice::Present (A CPU NAO SABE QUANDO QUE A GPU ACABOU DE REALIZAR O SEU TRABALHO.).
  • A GPU comeca a realizar as intrucões recebidas (como se estivesse em uma “outra thread =P”).

Esse assincronismo é completamente escondido do desenvolvedor (através do driver e do DirectX), porém em alguns momentos é interessante saber que ele existe. Por exemplo, devemos saber que ao ler algo da memória da placa de vídeo (para a RAM do PC) é algo bastante custoso, uma vez que a CPU e a GPU deveram se alinhar por uns instante e isto implica em uma das duas ter que esperar a outra. (como a GPU é um processador altamente pipelinezado, isso mata a performance (pipeline Stall)).

Ao usar um software de Profiler de CPU (como o nprof — que virou slimtune agora)  em uma aplicação XNA , tem-se apenas os seguintes tempos:

  • Um tempo gasto na função Game.Update
  • Um tempo gasto na função Game.Draw
  • Um tempo gasto na função GraphicsDevice.Present (wrapper para o Present nativo)
  • Um tempo gasto na IDirect3DDevice::Present (DirectX API convertendo as chamadas para a “linguagem” do driver)
  • Um tempo gasto no driver (ATI ou NVIDIA, driver convertendo as chamadas para a “linguagem” da GPU)

Para descobrir o tempo gasto na GPU devemos usar profilers de GPU, os mais usados são:

  • NVidia PerfHud (Testei com XNA 3.1 e DirectX 9c, não funciona com DirectX 11 — testado). Usei bastante esse profiler, gosto da sua interface simples e das features que ele apresenta. para usa-lo com todas as suas funcionalidades sua placa tem que ser suportada pelo release que voce baixar (ATI sem chance …). É possivel extrair dados como carga de trabalho da unidade de Pixel Shader, Vertex Shader, memória total utilizada, fluxo de dados trocados com a CPU … entre outros. Deve-se alterar a aplicação (inserir umas poucas linhas) para poder integra-la com o PerfHud
  • Nvidia Parallel Nsight (Testei com DirectX 11). Usei um pouco e gostei muito, principalmente da integração com o Visual Studio 2010 que facilita absurdamente as coisas. Contem praticamente todas as ferramentas do anterior e ainda suporta CUDA.
  • Microsoft PIX, vem junto com a DirectX SDK, usei com XNA e DirectX 9c. Nao tem a mesma profundidade que os dois citados acima, porém contem uma funcionalidade muito interessante. A capacidade de salvar todas as chamadas DirectX feitas em um frame e depois analisa-las. ÓTIMO para debugar shaders (verificar qual parâmetros estão entrando no vertex shader, quais estão saindo para o rasterizador …)

GPU e CPU Bounding

Quando um frame demora muito tempo para ser desenhado (se diversos comandos complexos forem enviados para a GPU) e o método Present for chamado antes q a GPU consiga terminar o frame anterior, a CPU terá que esperar antes de poder enviar as informações deste novo frame para a placa de vídeo. Esta situação é conhecida como GPU Bound, e está ilustrada na figura abaixo (o vermelho representa o tempo em que a CPU fica sem fazer “nada”).

GPU BOUNDED

O caso análogo a este acontece quando o CPU demora mais que a GPU em um frame (e esta ultima fica um tempo sem realizar nenhuma função), neste caso dizemos que a aplicação esta CPU Bound. É de responsabilidade da equipe de desenvolvimento equilibrar o consumo de GPU e CPU em uma aplicação, a ideia é nunca estar Bounded em nenhum dos dois (mais isso é muito difícil devido a dinamicidade de um jogo)

Curiosidade I: Um fator importante que pode afetar o tempo gasto na função GraphicsDevice.Present é o Vsync (sincronizar a taxa de desenho da GPU com o frameRate do monitor), uma vez que emalguns momentos o método deverá esperar o tempo de refresh do monitor antes de enviar o frame.

Curiosidade II: No DirectX 11 é possivel criar múltiplas threads na GPU (o funcionamento e a organizacão é semelhante ao Cuda, dá pra aprender bastante rápido como a mágica funciona), o assíncronismo é levado um passo adiante.
